Assignment Specification:

In this programming assignment we will develop our own multi-texture shader/renderable support for the game engine. My suggestion is to follow the engine architecture and define:

         multi_texture_fs.glsl + multi_texture_vs.glsl: that support at least three textures, a basic texture that covers the entire Renderable, and, two additional texture that the client can control. Definition of control to follow.

         MultiTextureShader: to interface and control the GLSL shaders

         MultiTextureRenderable: to allow clients to create and manipulate different instances of multi-texturing renderables.

 

Multi-Texturing Functionality:

The multi-texture system must support:

         At least three textures mapping on the same Renderable

o    First texture is similar to the texture on the existing TextureRenderable where the texture should cover the entire Renderable.

o    Second and Third textures: we will refer to these two as the effect textures. The placement of these two textures are under client control and can be mapped to any location on the Renderable with any size and rotation. More specifically, the client can control the location, size, rotation of each of the effective textures independently. This means, setting of one texture will not affect the other texture.

  For example, increasing and decreasing the size of one effect texture should not affect the size of the other effect texture.

  You must support the setting/getting of each effect textures: Position, Size, and Rotation

         Effect Texture drawing mode: Each effect texture must support four drawing modes:

o    None: The effect texture is hidden (not drawn)

o    Transparent: Based on the textures alpha value, turn the texture region completely transparent creating a window through the underlying TextureRenderable

o    Override: Draw over the underlying TextureRenderable

o    Blend: Blend the effect texture with the underlying TextureRenderable similar to the following blending function. The following listing is pseudo fragment shader code, you can adjust or modify this function as long as it achieves a similar blending effect. Note, when in the blending mode, the client has control over the blendWeight (a float):

 

if (effectTexture.a > 0.9) { // almost opaque

vec3 blendColor = (1.0 - blendWeight) * vec3(effectTexture);

color.rgb = current.rgb * blendWeight + blendColor;

color.a = length(blendColor) + (1.0-blendWeight) * current.a;

}

// effectTexture: is pixel read from the effect texture

// color: is the current computed color value, e.g., from TextureRenderables texture

 

         Testing: You must create a test scene that demonstrate the new functionality is not interfering with other existing functions.

o    Define a SpriteRenderable: to demonstrate that it is functioning, and, for demonstrating transparency of the effect texture. You must support WASD moving of this object. This object must be drawn before the multi-texturing renderables such that it will appear under the multi-texturing renderiable.

o    Define a TextureRenderable: to demonstrate that this is functioning, and, for demonstrating transparency of the effect texture. Once again, t his object must be drawn before the multi-texturing renderables such that it will appear under the multi-texturing renderiable.

o    Support at least three separate instances of the MultiTextureRenderable where the user can independently control the effect textures position, size, and rotation. For convenience of testing, you must support

  Ability to select one of the MultiTextureRenderable (in my implementation, the Z, X, and C keys).

  For the selected MultiTextureRenderable: Please make sure the key bindings for the manipulation are identical to the following (for my sanity when testing!)

         Key 0: binds the following to the Renderable transform

o    Arrow: moves the entire Renderable

o    Shift+Arrow: changes the size of the entire Renderable

o    Control+Left/Right-Arrow: rotates the entire Renderable

         Key 1 and 2: binds the following to the corresponding effect textures

o    Arrow: moves the selected effect texture

o    Shift+Arrow: changes the size of the selected effect texture

o    Control+Left/Right-Arrow: rotates of the selected effect texture with respect to the center of the texture

o    Keys: 3, 4, 5, 6: changes selected effect texture mode to

  None, Transparent, Override, and Blend

o    Keys N/M: increases/decreases the blend weight

o    Status display: at all times you need to echo the currently selected MultiTextureRenderable object, which texture is currently being manipulated, the textures position/size/rotation, and the current effect mode.

5.       Hint on aspect ratio of texture space for effect texture.

a.        Remember, the Renderable has WxH.

b.       By default TextureRenderable maps UV 0 to 1 onto the WxH.

c.        Now imagine, if your Renderable is 20x10, and you map this 20x10 to 1x1 texture coordinate, then, the texture will show up stretched.

d.       For effect texture, since you must support rotation, you must make sure the texture space aspect ratio obeys the aspect ratio of the Renderable, otherwise, the effect texture will appear distorted when rotated (this is exactly the same as our initial NDC issue).

e.        So, in the case of a 20x10 Renderable, you want the effect texture space to be 2x1 (instead of 1x1), and to ensure proper centering, you want to make sure the range of U is defined to be left: -1, right: 1 (so width is 2), V in this case is still range from 0 to 1.
